# Slack × A–D Knowledge Loop Tracker

毎日のSlack会話から「知識生産ループ（未知→観測→検証→資産化）」の回転を抽出し、チームの改善ループを底上げするための軽量フレームワーク。

## 概要

> **目的**: 人を評価することではなく、**チームの学習・検証・再利用の能力（KPL: Knowledge Production Loop）を底上げ**すること。

- ✅ Slackの日次スレッドで A–D タグを混ぜて会話
- ✅ AIが A–D イベントを抽出して日次でシートに整理
- ✅ 日次で「新規A」「A→B」「滞留A中央値」を確認
- ✅ チームの重心は個人点数ではなく、状態遷移比率の中央値で追う

## A–D 状態遷移モデル

| 状態 | 名称 | 定義 |
|:---:|:---|:---|
| **A** | Unknown | 未知・不確実性・懸念・仮説が「新規に言語化された」状態 |
| **B** | Observation | 未知を潰すための観測（テスト/計測/確認）に変換された状態 |
| **C** | Verified | 観測により仮説が「潰れた/強まった/否定された」状態 |
| **D** | Reused | 次回も回る形になった状態（自動化・テンプレ化） |

## Quick Start

```bash
# 1. Slackに日次スレッドを用意
# 2. 任意タグ A/B/C/D を周知（強制しない）
# 3. AI抽出を1週間回す（精度は気にしない）
# 4. Topic台帳を作成、週1回だけ重複マージ
# 5. 日次で「新規A」「A→B」「滞留A中央値」を確認
# 6. 月1回、ルールを1つだけ改善
```

## Slackテンプレ（任意）

```
- A: 今日の未知/不安/仮説
- B: 次に取る観測（テスト/計測/確認）
- C: 分かったこと（結果）
- D: 次回も回る形にしたこと（資産化）

（任意）
- 判断: 採用/不採用/保留
- 反証条件: これが起きたら間違い
```

## 日次で見る指標

1. **新規A件数** — 未知が表に出ているか
2. **A→B件数** — 未知が観測に変換されているか
3. **滞留A年齢中央値** — 未知が腐っていないか

## 注意事項

- 本フレームワークは**改善のため**。人事評価・査定に使うとゲーム化する
- Slackログには機密・個人情報が混ざりうる → 公開時は必ずマスキング
- AI抽出の誤判定は起きる → 精度より「ループが回ること」を優先
